# 널스넷 마이페이지 재설계 — 최종 실행 계획서

> **오너 확정 제약 (계획 본문보다 우선)**
> 1. **회원가입은 지금처럼 쉽게** — 이메일·비밀번호·회원유형 3가지만 받는다. 가입 폼에 항목을 추가하지 않는다.
> 2. **개인정보(이름·연락처)는 이력서를 쓸 때 받는다.** 따라서 간호사의 이름·연락처는 `resumes`가 원본이고,
>    `/mypage/account`(계정) 화면에는 간호사용 이름·연락처 입력을 두지 않는다(이력서로 링크만 건다).
>    병원 회원은 이력서를 쓰지 않으므로 담당자 이름·연락처만 계정 화면에서 `profiles`에 받는다.
> 3. 지원 버튼을 눌렀을 때 이력서가 없거나 이름·연락처가 비어 있으면 이력서 작성 화면으로 보낸다.

> 이 문서는 감사 결과(실제 코드·마이그레이션 확인)와 업계 조사, 그리고 3개 설계안 심사를 하나로 합친 **최종 계획**입니다. 코드는 없습니다. 위에서부터 순서대로 진행하시면 됩니다.

### 이력서 작성 부담 관리 (중요)

항목이 11개 → 약 30개로 늘어납니다. 그대로 두면 작성률이 떨어지므로 아래를 함께 넣습니다.

- **저장 시 필수는 6개만**: 이름 · 휴대폰 · 면허구분 · 신규/경력 · 가능한 근무형태 · 희망 근무지
- 나머지는 전부 선택 + **"이력서 완성도 %" 게이지**로 유도
- **기존 이력서 대응**: 이미 저장한 분들은 다음에 저장할 때 한 번만 보완하면 됩니다. 이름·휴대폰이 비어 있으면 **지원 버튼을 누를 때** 서버에서 막고 이력서 화면으로 안내합니다(현재 `applyToJob`은 이력서 행 존재만 확인).

## 5. DB 변경 목록

### 5-1. 0단계 (버그 지혈)

| 대상 | 변경 | 이유 |
|---|---|---|
| `resumes.is_public` | 기본값 `true` → **`false`** | 지금은 저장 코드가 항상 명시 저장해서 가려져 있지만, 다른 경로(시드·스크립트)로 행이 생기면 그 순간 이름·전화번호가 기본 공개됩니다. 한 줄입니다. |

나머지 0단계 항목은 **DB 변경 없이 코드 수정만**으로 끝납니다.

### 5-2. 1단계 (지원자 처리·통보)

| 대상 | 변경 | 이유 |
|---|---|---|
| `applications.status` | 체크 제약 확장 → `submitted / viewed / interview / accepted / rejected / withdrawn` | 면접 단계가 없어 합격·불합격 2택뿐. 지원 취소 상태도 없음 |
| `applications` | `hospital_note text` 추가 (담당자 메모) | 저장할 곳이 코드·DB 전체에 0곳 |
| `applications` | `interview_at timestamptz`, `status_note text` 추가 | "합격" 세 글자만으로는 언제 어디로 가야 할지 알 수 없음 |
| `applications` | `resume_snapshot jsonb`, `applicant_name text`, `applicant_phone text` 추가 | ① 이력서를 고치면 과거 지원서가 소급 변경됨(resumes가 1:1 구조) ② 이력서를 비공개로 돌려도 병원이 연락할 수 있어야 함 |
| `applications` **권한** | `revoke update ... from authenticated;` **먼저**, 그 다음 `grant update (status) to authenticated;` | ⚠️ **순서가 중요합니다.** Supabase 기본값이 테이블 전체 UPDATE 허용이라, 컬럼 grant만 추가하면 아무것도 좁아지지 않습니다. 이걸 안 하면 지원자가 `hospital_note`·`resume_snapshot`까지 조작할 수 있습니다 |
| `applications` **정책** | `applications_withdraw_own` 신설: `using(applicant_id = auth.uid())` + `with check(applicant_id = auth.uid() and status = 'withdrawn')` | UPDATE 정책은 OR로 합쳐지므로, 간호사는 이 경로로 **'withdrawn' 외의 값으로는 절대 못 바꿉니다.** 병원의 기존 정책은 영향 없음 |
| `applications` unique | `unique(job_id, applicant_id)`는 **유지**, 대신 재지원은 기존 행을 `submitted`로 되살림 | 삭제하면 병원 기록이 흔적 없이 사라집니다. `hasApplied`도 withdrawn을 제외하도록 함께 수정해야 합니다 |
| `saved_jobs` | `job_title text`, `hospital_name text`, `saved_deadline date` 추가 | 병원이 공고를 지우면 `on delete cascade`로 저장 행 자체가 사라집니다. 스냅샷이 있으면 목록에 남습니다 |
| **신규 테이블** `notifications` | `(id, profile_id, kind, title, body, link, is_read, created_at)` · RLS: 본인 select + is_read update | 합격/신규 지원 신호가 도달할 최소 경로 |
| `ad_orders.buyer_id` | `not null + on delete cascade` → **nullable + on delete set null** | ⚠️ 지금 상태로 회원 탈퇴를 만들면 **결제 기록이 함께 삭제됩니다.** 전자상거래법상 대금결제 기록 5년 보존 위반이고 포트원 대사도 불가능해집니다 |

### 남는 위험 (알고 계셔야 할 것)

1. **탈퇴하면 그 사람의 지원 기록이 병원 화면에서도 사라집니다.** `applications.applicant_id`가 `on delete cascade`입니다. 개인정보 파기 관점에서는 맞는 동작이지만, 면접까지 본 병원의 기록이 통보 없이 없어집니다. 탈퇴는 유예 없이 즉시 삭제이므로 **확인 문구 직접 입력**을 요구합니다.
2. **워크넷·공공데이터 공고(DB의 대부분)는 여전히 외부 사이트로 나가 지원 기록이 남지 않습니다.** 이 계획으로 해결되지 않습니다. 간호사의 지원 내역은 계속 반쪽입니다.
3. **이력서 항목이 늘면 작성률이 떨어집니다.** 필수 6개 + 완성도 게이지로 완화하지만, 지금의 "5분이면 끝나는 이력서"보다는 무겁습니다. 3단계 배포 후 이력서 저장 건수를 꼭 지켜보시고, 눈에 띄게 줄면 필수 항목을 더 줄이십시오.

# 부록 A. 최종 점검에서 나온 수정사항 (계획 본문보다 우선)

## 점검 1

계획대로 실행해도 두 불만은 **부분적으로만** 해소됩니다. 아래는 실제 코드·마이그레이션을 대조해 찾은 누락입니다.

## A. 계획대로 하면 막히거나 구멍이 나는 것 (치명)

- **applications의 INSERT 컬럼 권한이 계획에 없습니다.** 5-2는 `revoke update` → `grant update(status)`만 다룹니다. 그런데 `applications_insert_own`(20260625210000:25)의 with check는 `applicant_id`만 봅니다 → 간호사가 anon key로 PostgREST에 직접 `status='accepted'`로 insert하면 **병원 화면에 "합격"으로 뜹니다.** 1단계에서 `applicant_name/applicant_phone/resume_snapshot/hospital_note`가 추가되면 **가짜 이름·연락처 주입**까지 가능해집니다. 필요한 조치: `revoke insert on applications from authenticated` → `grant insert (job_id, message)`, insert 정책 with check에 `status='submitted'` 추가, 스냅샷 3컬럼은 서버 액션(admin) 또는 BEFORE INSERT 트리거로만 채우기.
- **"재지원은 기존 행을 submitted로 되살림"이 같은 표의 다른 두 줄과 충돌합니다.** `grant update(status)`만 주고 `applications_withdraw_own`의 with check를 `status='withdrawn'`으로 고정하면, 간호사는 **withdrawn → submitted로 되돌릴 수 없고**(병원 정책도 만족 못 함), 재지원 시 `message`·`resume_snapshot`·`applicant_phone`도 갱신 불가라 **낡은 스냅샷이 병원에 갑니다.** 재지원은 SECURITY DEFINER RPC나 admin 클라이언트 서버 액션으로 처리한다고 못 박아야 합니다.
- **"실제 펼친 1건만 열람 기록"을 `?open=` 링크로 하면 지금과 같은 함정에 다시 빠집니다.** 자동 열람이 admin 클라이언트로 되어 있는 이유가 바로 서버 컴포넌트 렌더 중 쓰기이기 때문입니다(`C:\dev\nurse-app\src\lib\data\applications.ts:63-69`). 링크(GET)로 펼치면 또 렌더 중 UPDATE가 필요합니다. **"지원자 열기"를 form(POST) 버튼으로 만들고, 액션에서 viewed 기록 후 `?open=`으로 redirect**하는 방식을 계획에 명시해야 합니다.

## B. 원래 불만이 그래도 남는 지점

- **병원 불만은 0단계에서 전혀 해소되지 않습니다.** 이름·휴대폰 필수화는 2단계, 지원 시 차단·스냅샷은 1단계인데, 0단계 "이력서 전 컬럼 조회"를 해도 `resumes.name/phone`은 지금도 선택 입력이라 **null이면 여전히 연락 불가**입니다. 최소한 "이름·휴대폰 없으면 지원 차단 + 이력서로 안내"는 0단계로 당겨야 0단계 마무리 문구("병원은 인재검색과 같은 수준으로 본다")가 사실이 됩니다.
- **예비 연락 수단이 끝까지 없습니다.** 병원은 `profiles.email`을 읽을 수 없고(20260625161000), 이력서 `email` 컬럼은 3단계입니다. 1단계 스냅샷 저장 시 `applicant_email`(profiles.email 복사) 한 컬럼을 같이 넣으면 비용 0인데 빠져 있습니다. 지금 계획으로는 **전화를 안 받으면 접촉 경로가 0**입니다.
- **병원이 아무 조치도 안 할 때의 처리가 계획 전체에 없습니다.** 상태가 submitted/viewed로 영원히 멈춰도 간호사에게는 아무 일도 일어나지 않습니다. "지원 후 N일 경과" 표시, 무응답 자동 만료, 병원 응답률/평균 응답일 중 최소 하나가 필요합니다. "합격했는지 모른다"의 절반은 이게 원인입니다.
- **담당자 연락처 노출 규칙이 기존 화면과 모순됩니다.** 공고 상세는 **로그인만 하면** `manager_phone`을 그대로 보여줍니다(`src/app/jobs/page.tsx:265-286`). 그런데 계획은 지원 내역에서 합격·면접일 때만 노출합니다 — 이미 공개된 정보를 숨기는 것이라 실익이 없고, 정작 필요한 시점(지원 직후 문의)에 막힙니다.
- **워크넷/외부 공고에 대한 대안이 "위험 2번" 한 줄뿐입니다.** DB의 절대다수가 external_url 공고라, 이 계획을 전부 끝내도 간호사의 지원 내역·합격 여부는 direct 공고에서만 존재합니다. 저렴한 완화책(`saved_jobs.applied_at` 또는 "지원함으로 표시" 셀프 기록 + 지원 내역에 외부 지원 섹션)이 0~1단계에 들어갈 만한데 후보로도 없습니다.

## B. 실행하면 깨지는 충돌 (수정 필요)

**1. [치명] `grant update (status)`만 재부여하면 병원 메모·면접일 저장이 전부 실패**
`updateApplicationStatus`는 **사용자 세션**(`createClient()`)으로 씁니다. 5-2대로 `revoke update; grant update (status)`를 하면 같은 1단계에서 추가하는 `hospital_note`·`interview_at`·`status_note`를 병원이 못 씁니다(permission denied). 그렇다고 컬럼을 더 열면, 간호사가 `status='withdrawn'`과 `hospital_note`를 **한 번에** 보내면 withdraw 정책 with-check를 통과해 병원 메모를 덮어씁니다.
→ 해결: `grant update (status)`는 그대로 두고, **병원 쪽 쓰기를 전부 admin 클라이언트 + 소유자 검증 코드로** 옮기세요(이미 `ownedJobHospital()` 패턴 있음).

**2. [높음] 재지원 경로가 자기 정책에 막힘**
"재지원은 기존 행을 `submitted`로 되살림"인데, 간호사의 withdraw 정책 with-check가 `status='withdrawn'`만 허용합니다. `withdrawn → submitted`는 간호사 세션으로 **불가능**합니다. 재지원도 admin 경유여야 합니다. (`resume_snapshot`·`applicant_name/phone`도 INSERT 권한은 회수 대상이 아니라 간호사가 위조 가능 → 스냅샷은 service_role이 써야 신뢰 가능)

**3. [높음] 2단계 "회원 유형(간호사↔병원) 변경"이 리뷰 방어를 되돌림**
`20260723090000`은 병원 계정의 자작 리뷰를 막으려고 `reviews_insert_own`에 `role='nurse'`를 박아 넣었습니다. 셀프 역할 전환을 열면 **병원 → 간호사 전환 → 자기 병원 5점 리뷰 → 다시 병원**이 가능해집니다. (`profiles.role`은 grant에서 빠져 있어 admin 경유는 맞지만, 그게 방어가 되진 않습니다.)
→ 전환에 쿨다운/1회 제한을 걸거나, 최소한 전환 시 작성 리뷰를 숨김 처리하세요.

**4. [중간] `/jobs/[id]`의 "마감 배너"와 "목록과 같은 노출 규칙"이 서로 모순**
`jobs_select_open`은 `status='open'`만 열어줍니다. 마감 공고에 "마감된 공고입니다" 배너를 띄우려면 **admin 조회가 필수**인데, 이는 계획서 자신의 함정표("admin 조회를 무제한으로 열지 마라")와 충돌합니다. `toggleSaveJob`이 job_id를 검증하지 않으므로 무제한 열면 draft/hidden 공고가 새어 나갑니다.
→ 규칙 명시: **"내 applications 또는 saved_jobs에 있는 id"일 때만** admin 폴백, 그 외엔 404.

## C. 단계 순서상의 구멍 (작지만 실제)
